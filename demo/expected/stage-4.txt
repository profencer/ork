# Change plan — client-side request rate limiter for the Anthropic TypeScript SDK

## Overview
Add an opt-in `requestsPerMinute` option to the `Anthropic` client constructor.
When set, every outgoing request goes through a token-bucket gate before the
underlying `fetch` is called; requests that exceed the budget are either
queued (default) or rejected with `RateLimitExceededError`.

## Per-repository changes
### anthropic-sdk-typescript
- `src/client.ts` — extend `ClientOptions` with `requestsPerMinute?: number`
  and `rateLimitMode?: 'queue' | 'reject'`.
- `src/internal/rate-limiter.ts` (new) — token-bucket implementation.
- `src/core.ts` — wrap the request pipeline so the limiter runs before `fetch`.
- `README.md` — document the new options under "Client options".
- `tests/rate-limiter.test.ts` (new) — unit tests covering queue + reject modes.

## Cross-cutting concerns
- The limiter must be a no-op when `requestsPerMinute` is unset to preserve
  backward compatibility.
- Streaming requests (`messages.stream`) count as a single budgeted call.

## Suggested ordering
1. Land `internal/rate-limiter.ts` + unit tests in isolation.
2. Wire the limiter into `core.ts`'s request pipeline behind the new option.
3. Surface the option in `ClientOptions` and the README.

## Risks and open questions
- Queue mode interacts with the SDK's existing retry policy; need to decide
  whether retries count against the rate budget.
- No clean place to surface budget metrics — possibly out of scope.
