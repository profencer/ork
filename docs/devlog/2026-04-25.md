### Bugs surfaced by dogfooding (and fixed)

Each of these would have been invisible without ADR-0012's
multi-step, multi-LLM workflow run. Order is roughly the order they
fell out of stage-4.

1. **Minimax `Authorization` header was a bare key, not `Bearer …`.**
   Documentation-only fix in `.env.example` plus a `MINIMAX_API_KEY`
   format note. Caught the first 401 of the day.
2. **`for_each` resolved template not a JSON array.** Symptom of two
   different upstream bugs (3 + 4 below); the template engine itself
   was correct. Note in the incident file is what kept us from
   "fixing" it in the wrong layer.
3. **`git clone` race condition in `crates/ork-integrations/src/workspace.rs`.**
   Concurrent worker fan-out on the same repo path raced
   `git clone --depth 1` and either left a half-formed `.git/` or
   blew up with FETCH_HEAD writes. Fixed with an
   `Arc<StdMutex<HashMap<PathBuf, Arc<TokioMutex<()>>>>>` keyed on
   the workspace path; regression in
   `crates/ork-integrations/tests/workspace_concurrent.rs` (new). The
   integration test only runs reliably outside the workspace sandbox
   (`git init` needs unrestricted access to `.git/hooks/`); it's
   in CI but excluded from the in-sandbox `cargo test --workspace`
   verification on this machine.
4. **`peer_*` tool dispatch surfaced as fatal `Workflow` error
   instead of recoverable `Validation`.** A `peer_<self>_*` call
   tripped the cycle detector at depth 1 — but more fundamentally,
   the agent's tool catalog was advertising every agent's own peer
   skills back to itself. Two-part fix: (a)
   `ToolCatalogBuilder::for_agent` filters `peer_<self>_*` out of
   the calling agent's catalog so the LLM never sees self as a
   peer; (b) `AgentContext::child_for_delegation` cycle/depth
   rejections moved from `OrkError::Workflow` (fatal per ADR-0010)
   to `OrkError::Validation` (recoverable, returned as a tool
   result). Regressions:
   `crates/ork-agents/src/tool_catalog.rs::builder_does_not_advertise_self_peer_tools`
   and updates to `crates/ork-core/tests/delegation_step_cycle_cap.rs`.
5. **`agent_call` validation rejected calls with empty `prompt`.**
   The composite tool executor passed a `peer_*` payload into
   `AgentCallToolExecutor::execute` without normalising the shape;
   the `prompt` field was missing rather than empty. Fix +
   regression in `crates/ork-integrations/tests/agent_call_tool.rs`.
6. **`a2a_tasks_parent_task_id_fkey` violation on delegation.**
   `delegate_to:` and `peer_*` child tasks tried to insert before
   the parent task's row existed. Fixed by inserting parent
   `a2a_tasks` rows in `WorkflowEngine::execute_agent_step` /
   `execute_delegated_call` before the agent runs. Regression:
   `crates/ork-core/tests/engine_persists_parent_task.rs` (new),
   which uses an `FkEnforcingTaskRepo` shim so the test asserts on
   FK semantics without a Postgres dependency.
7. **Workflow cascaded past a failed step on `depends_on` edges.**
   The compiler emitted `EdgeCondition::Always` for every
   `depends_on` directive, so a failed parent fired its children
   with bogus `{{parent.output}}` and the polling loop sat at
   `running` until timeout. Changed to `EdgeCondition::OnPass`;
   authors who want a fan-out-on-failure path now declare it
   explicitly via `condition.on_fail`. Regression:
   `crates/ork-core/tests/engine_failed_step_does_not_cascade.rs`
   (new, two cases: linear-chain termination and
   explicit-`on_fail` still routes correctly).
8. **Engine step-lifecycle logs lacked `run_id`.** With the cascade
   above, `tail -F demo/logs/ork-api.log | jq 'select(.fields.run_id == "...")'`
   returned only the two service-layer create/start lines and missed
   every per-step event. Threaded `run_id` (and existing `step_id`)
   through every relevant `info!`/`warn!` in
   `crates/ork-core/src/workflow/engine.rs`. Verified by reading the
   live log of the very next demo run.
9. **`OpenAiCompatibleProvider` had no retry on transient
   stream/connect failures.** Initial attempt at a fix only covered
   the mid-stream `bytes_stream` poll path; a follow-up demo run
   surfaced the symmetric initial-send class
   (`request failed: error sending request for url (...)`, i.e.
   reqwest's `send().await` failing before any response head
   arrives). Final shape: `send_chat_stream_attempt` returns an
   internal `AttemptError::{Transient, Fatal}` enum (transient =
   reqwest transport error or 5xx, fatal = 4xx); `chat_stream`
   retries once on transient errors **at both layers** but shares a
   single `STREAM_MAX_ATTEMPTS = 2` budget so the worst case is two
   HTTP requests per `chat_stream` call. Mid-stream retries only
   fire if no SSE event has been forwarded to the consumer (replays
   would corrupt the output). Regression:
   `crates/ork-llm/tests/openai_compatible_stream_retry.rs` (new,
   four cases). The test uses a hand-rolled `tokio::net::TcpListener`
   to simulate precise mid-stream truncation and immediate-after-
   accept connection drop because `wiremock` cannot reach those
   primitives.
10. **`jq fromdateiso8601` rejected fractional-second timestamps.**
    Engine emits `2026-04-25T12:34:48.547477Z`; jq builtin only
    handles `%Y-%m-%dT%H:%M:%SZ`. Stripped the fractional component
    inside `fmt_steps_with_durations` in
    [`demo/scripts/stage-4-workflow.sh`](../../demo/scripts/stage-4-workflow.sh).
    Same edit fixed an earlier infinite-recursion bug where a jq
    binding had been named `now`, shadowing the builtin.
11. **Demo polling timeout clipped legitimate completions by
    seconds.** `WORKFLOW_TIMEOUT_SECS` defaulted to 180; a healthy
    end-to-end stage-4 against Minimax took 187s in one observed run
    (5 LLM-driven steps, `research_repos` alone burns 60–90s when
    its tool loop is busy). Bumped the default to 600s. Env var
    override still honoured for operators with a faster local
    catalog.

  flaky in the in-sandbox `cargo test --workspace` run** because
  the sandbox refuses `git init`'s writes to `.git/hooks/`. Passes
  cleanly with `required_permissions: ["all"]`. Either gate the
  test on a CI-only env var or move it to a separate `--features
  unsandboxed-git` target — pick one and document it in
  [`docs/operations/`](../operations/).
- **`LlmRouter::tenant_provider_cache` has no upper bound.** Flagged
  in the ADR-0012 reviewer Minor #3 as a follow-up; current
  `RwLock<HashMap<...>>` grows linearly with `(#tenants × #providers
  per tenant)`. Fine for any near-term deployment, but worth a TODO
  comment with an LRU eviction owner before the long-running ork-api
  process starts collecting tenants in production.
- **`AgentConfig.model` precedence comment in
  `crates/ork-llm/src/router.rs`** still half-truthful in places
  outside the path I touched; sweep it for correctness in the next
  PR-split pass.
