## Top-line read

**The pivot is not a pivot. It is an addition.** All 23 prior ADRs (the
A2A enterprise-platform pitch) are still in place; 18 more have been
bolted on (the autonomous-coding-agent pitch). The director said
*"too early"* and the response was *"more scope, more layers, more
depth."* That makes the readiness gap larger, not smaller.

Concretely: of the 44 numbered ADRs, **12 are shipped** —
Implemented (0003, 0013, 0015) plus Accepted-and-implemented (0001,
0002, 0006, 0007, 0010, 0011, 0012, 0016, 0017). That is real
substrate: Agent port, peer delegation, remote A2A client, MCP plane,
native LLM tool calling, multi-LLM router, generic gateways, dynamic
embeds, artifacts, Web UI — more shipped surface than CrewAI had at
its Series A. A 9-stage end-to-end demo (`demo/scripts/stage-*.sh`)
runs on this substrate today, including a federated LangGraph peer.

What is *not* shipped is the rest: 0004 transport, 0005 discovery,
0008 server endpoints, 0009 push, 0018 DAG enhancements remain
`Proposed` from earlier phases, and the entire 0019–0045 batch is
`Proposed`. The new 18-ADR batch lands a 17-deep transitive dependency
chain on top of that backlog. ADR 0045 (the headline coding-team
demo) cannot be built until 0028, 0029, 0030, 0031, 0033, 0037, 0038,
0040, 0041, 0042, 0043, 0044 are code. None of them are yet. That
specific batch is a wishlist tree, even if the platform underneath it
is real.

## What is genuinely good

The **writing quality of 0028–0045 is unusually high for ADRs**.
Boundaries with existing ADRs are explicit (0033 vs 0045 table, 0044
vs 0031 table, 0045 vs 0044 boundary). Schemas are versioned by URL.
The decomposition (verbs in 0028–0030, nouns in 0033/0042, composition
in 0044/0045) is internally coherent. Rare to see this much rigor.

**Individually well-scoped, defensible ADRs:**

- [0028](../docs/adrs/0028-shell-executor-and-test-runners.md) — sharp argument for
  in-process sandboxing rather than `mcp:shell`. Genuinely necessary.
- [0029](../docs/adrs/0029-workspace-file-editor.md), [0030](../docs/adrs/0030-git-operations.md) — minimal, ship-it shape.
- [0034](../docs/adrs/0034-per-model-capability-profiles.md) — names a real tuning gap
  (frontier vs 7B) precisely.
- [0040](../docs/adrs/0040-repo-map.md) — cheap, high-leverage, Aider-validated prior art.
- [0019](../docs/adrs/0019-scheduled-tasks.md) — small, finishable, value-clear; should
  ship.

## What is broken

**1. Sequenced dependency depth is fictional.** ADR 0045 reads as if
0033/0038/0040/0041/0042/0043/0044 already exist. They are all
`Proposed`, three of them >1000 lines, none with code. The "compose
seven other ADRs" pattern means there is exactly one valid build order
and any single ADR slipping blocks the demo. That is the opposite of
the work loop in [AGENTS.md](../AGENTS.md) §4 ("one ADR at a time"). The
project broke its own rule.

**2. No metric of done that maps to the director's question.** The
director asked "is this real for our org?" Re-read the new ADRs: there
are zero numbers about *defects caught*, *PRs landed by ork*,
*cycle-time reduction on real work*, *cost per autonomous fix*. ADR
0045's success criterion is "the team ships a green commit on a feature
branch from a clean checkout" — that is *liveness*, not *value*. Cursor
and Claude Code already pass that bar today on every developer's
laptop.

**3. The differentiator argument is a feature, not a use case.** 0038
and 0045 both lean on "single-process harnesses cannot run plan
verification across vendors/tenants/processes." True. But name the
*customer* who will pay for that. Most enterprise coding-agent buyers
want fewer, cheaper, auditable agents — not a multi-agent verification
mesh that triples token spend per task. The competitive moat is
asserted, never priced.

**4. The eval/regression gap is the credibility hole.** The repo's own
[todos.md](../todos.md) calls it out: "Nothing in the ADR set addresses 'how
do I prove this workflow didn't regress.'" That is exactly what the
director was saying. There are 27 architecture ADRs and zero ADR for
"how do we measure whether the agent does what we claim." LangSmith-
style dataset + replay is not optional for enterprise; it is the
*first* thing a buyer asks for.

**5. The presentation oversells maturity in four specific places.**
[build_managers_deck.py:302–339](../build_managers_deck.py#L302-L339)
mostly reflects code that exists — Agent port, generic gateways,
multi-LLM router, MCP plane, embeds, artifacts, Web UI, federation
are all shipped. The four lines that are *not* honest are: the
A2A-stack bullet that includes `push` (0009 is Proposed), the
"Hybrid transport" claim citing 0004 (Proposed), the workflow-engine
bullet citing 0018 (Proposed), and the A2A-server-endpoints citation
0008 (Proposed). A director who reads the repo will see those four
gaps and discount the rest. Five-minute fix.

**6. Multi-agent fan-out without a token budget.** ADR 0044 + 0045
dispatch N parallel sub-agents, each with their own LLM, each with
plan verification across M peers (0038), each with constrained
decoding (0035), each with repo map and LSP context priming. That is
easily 10–30× the token cost of a solo Aider run. There is **no cost
model** in 0045 and no `Open question` flagging this. Enterprises
track $/PR; this design has no answer for that question.

**7. The "team" in "coding team" is fictional.** Who is the human
supplying prompts to architect, executor, reviewer, tester? If it is
one developer, multi-agent overhead pays for single-agent throughput.
If it is "the org," there needs to be an ADR for *who decides what the
team works on* — which is the Devin pitch and Devin has not made that
work either.

**8. Pivot timing is wrong.** April 2026: Cursor at scale, Claude Code
shipped, opencode and Cline established, Devin recovering from honesty
issues, Copilot Workspace mainstream. The autonomous-coding-agent
space is the **most contested AI surface that exists**, and a single
team in Rust ADR-driven mode is not going to outrun it. The original
"A2A enterprise mesh" pitch had the virtue of being boring — there is
real demand for it (because Solace is shipped to large orgs and people
want off it). The new pitch puts ork head-to-head with $1B-funded
products.

## What the director probably actually meant

"Too early state of AI adoption" rarely means "AI is not real." With
12 ADRs shipped and a 9-stage demo running, it almost certainly does
not mean "your code does not exist." More plausible shapes:

- *No internal user base in this org that will tolerate v1 quirks.*
- *Adjacent platforms (data, security, IDP) are also pre-pilot; ork's
  Kong+Kafka+Postgres assumptions need cooperation that is not formed
  yet.*
- *The deck's eight enterprise-requirements story is generic; he heard
  "I do not know what this is *for* in our org."*
- *Compliance/security will not approve autonomous code-writing agents
  touching prod repos this year.*

None of those are answered by 18 more ADRs. They are answered by **one
internal workflow ported onto the substrate that already runs, with
measurable time saved**. The substrate is in place; the new ADRs do
not produce that artifact, and the existing demo is not yet aimed at a
real internal use case.

## What I would do this week

1. **Fix the four false Maturity-slide lines** (push / 0004 / 0018 /
   0008). Five-minute edit; removes the only legitimate overselling
   complaint.
2. **Add a "что уже работает на демо" slide.** Stages 1–9, what each
   proves, screenshot of stage-9 federation. That is the direct
   answer to "too early."
3. **Pick one *internal* workflow to port** onto what already ships.
   Not a coding agent, not a crew demo — a workflow someone in this
   org would use weekly. The substrate is rich enough; this does not
   require new ADRs.
4. **Park 0036, 0039, 0042–0045** until 0028–0033 are real *or* the
   internal pilot tells you which ones it actually needs. The
   shipped substrate already exceeds CrewAI's OSS surface; expanding
   the spec before the pilot informs it is the wrong direction.
5. **Write the missing ADR**: evaluation/regression harness. Smallest
   scope: a test corpus of (input, expected-shape) pairs, a
   `cargo run --example replay` harness, a CI job. This is the single
   artifact that flips "too early" to "we have receipts."
6. **Rebuild the differentiator slide.** "Open A2A, no lock-in, Rust"
   are not buying decisions. "Runs on-prem inside your existing
   Kafka+Postgres footprint, with audit trail, without sending source
   code to OpenAI/Anthropic" is. Lead with that, with a cost number
   attached.
7. **Drop the autonomous-team headline until solo works.** ADR 0045
   is the most ambitious thing in this repo and the one with the
   weakest evidence base. It will look better proposed in October
   than rejected in May.

## On the emotional side

Two things worth separating:

- The presentation went badly because of *positioning and maturity
  claims*, not because the architecture is bad. The architecture is,
  frankly, the strongest part of this project.
- Pivoting in reaction to a bad meeting is the most common
  engineering-leader trap. The 18 new ADRs were probably written
  under "I need to prove this is more than a SAM port" pressure. They
  prove rigor; they do not prove value. A second meeting walking back
  to *"give me 8 weeks, here is the one thing I will demo"* will land
  better than 18 ADRs.

The project is not lost. 14k lines of high-quality architecture
landed in a week. That energy redirected into shipping 0019 + a real
`solo_coder` demo + an eval harness ADR is a different conversation
with the director by July.
